## Step 5 — Communication Protocol(整個 session 適用)

### Language
- 用**繁體中文**回覆(team primary language)
- Code comment + commit message 用英文

### Citation
- 引用 spec 必須標 section,e.g. `(per architecture.md §3.5)` / `(per CLAUDE.md §4.1)`
- 引用 OQ 必須標 ID,e.g. `(depends on OQ-Q3)`

### Tone
- **Direct**:避免 hedging(「可能...或者...」)。有判斷直接講 + 講理由
- **No flattery**:唔好「好嘅,呢個係好問題!」呢類開場
- **Confirm-first on substantive work**:任何寫 code、改 file、加 dependency,都要先得 user approve
- **Surface concerns early**:發覺 spec 不一致 / OQ default 唔合理 / Strict Mode trigger,**第一句就講**,唔好 bury

### Strict Mode Triggers(任何時候撞到呢啲 stop and ask)
- H1 Architectural change(spec §3 / §4 嘅 component / vendor / storage layout / multi-KB rule / view layout philosophy)
- H2 Add vendor / dependency 唔在 spec §3.2 lock 範圍
- H3 Use Dify reference 而想 copy code
- H4 Touch Tier 2 feature(GraphRAG / multi-agent / multi-tenancy / multimodal)
- H5 Security / privacy(log secret / commit credential)
- H6 Critical module 寫 code 唔同步寫 test

### When in Doubt
按 CLAUDE.md §11 嘅 default behavior table:
- Spec 同 your idea 衝突 → Spec wins,raise + ask
- Spec 缺 detail → Ask,don't guess
- 兩種 implementation 都 reasonable → 揀更接近現有 pattern 嘅
- Tier 1 / Tier 2 邊界模糊 → Default Tier 2(out of scope),ask
