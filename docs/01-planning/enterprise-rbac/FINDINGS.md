- 進階授權:**`groups=0`、`kb_acl=0`** —— per-KB 授權 + 群組**從未被實際使用過**

## 3. 根因分析

> **唔係規劃問題,亦唔係進度問題** —— 規劃(ADR-0027)同 W24c 交付都係乾淨完整嘅。

- **初判已糾正**:曾判斷「兩套角色定義並存技術債」,但 `git show HEAD:backend/api/schemas/rbac.py` 證實 HEAD 係**乾淨四級**;所見「三級 viewer」係**本機 / OneDrive 陳舊版本 + 後端跑舊碼**,非 HEAD 真實狀態。
- **真因 = 運行環境 / 檔案同步落差**:後端服務未重啟更新(跑舊碼)+ 本機 OneDrive 同步異常(同另一 session `W87-onedrive-path-migration` 同源)+ 既有債未收尾(首位管理員 bootstrap / 前端硬編 / 寫操作佔位)。
- **含意**:需要**基礎校正(P0)**令地基活起來,**非重做**;但校正之前,系統實際上不可用。

## 4. 缺口分析(對標企業級)

對標成熟企業權限系統(Microsoft Purview / SharePoint、Confluence、Glean):

| 缺口 | 重要性 | 現狀 |
|---|---|---|
| **檢索層文件 / 段落級存取控制** | 🔴 **最關鍵** | ❌ 現只到知識庫層 |
| 群組成員同步 + 繼承存取 | 🟠 高 | ❌ 成員同步未做 |
| 真實 Microsoft SSO 接駁 | 🟠 高 | ❌ 仍 mock |
| 管理權分級(職責分立) | 🟡 中 | ❌ 單一管理員全權 |
| 存取覆核 / 生命週期 | 🟡 中 | ❌ 未做 |
| 屬性式權限 / 集中政策引擎 | ⚪ 規模化才需 | ❌ 未做 |

> **為何檢索層安全最關鍵**:企業知識庫含不同敏感度文件,AI 問答若喺檢索階段撈到無權內容,會直接摘要呈現 → 資料外洩。故存取控制必須喺「檢索之前 / 之中」執行(LLM 係 confused deputy)。**此項屬正式上線先決條件,非 Tier 2 後續。** Azure AI Search S1 原生支援此類 `filter`,唔撞 vendor lock。

## 5. 改造性質(補充擴充,非重做)

- **唔需要推倒重做** —— W24c 係合理企業級基礎(160 測試為證),整體係「補充 + 往下延伸 + 加一個檢索安全層」。
- **一次性成本**:檢索索引需重建一次(資料遷移,非程式重寫)、需小心改 AI 檢索主流程(勿影響 W43–W85 校好嘅問答品質)、`kb_acl` 結構需演進。
- **唯一可能部分重寫** = 硬編權限矩陣 → 政策引擎(僅 P6 規模化才需,短期不掂)。
