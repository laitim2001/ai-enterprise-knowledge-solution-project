②RAG 查詢流程(核心功能:問答)

由用戶提問到答案渲染共 10 步。全程 Langfuse trace;每階段旋鈕受三層配置(圖 ⑦)控制。security trimming 內嵌喺檢索請求內,唔係事後過濾(ADR-0066)。

③文件 Ingestion 流程(核心功能:入庫)

由上載到可檢索共 10 步。重點:第 4-5 步「文件畫像 → preset 自動配對」係 ADR-0056 per-KB 可調配置 vision 嘅落地 — 唔同類型文件自動攞到度身訂做嘅 recall 配置;第 9 步同時寫入 security 欄位,令權限係「入庫時已收斂」而非查詢時補救。

④圖文還原機制(§15 北極星,核心功能:answer 品質)

準則:答案要忠實還原原文檔本來嘅圖文關係 — 原文某段有圖跟隨,recall 嗰段時圖就還原到對應位置。以下四層機制由粗到細逐層落地,疊加運作:

A

全圖召回地基(W59–W68,ADR-0054)

dedup-before-cap + 鄰近圖收割 + 錨點冗餘 — image-recall 由 0.574 提升到約 1.0(九 query 全 1.0 可達,precision 0.988)。成功定義 = 全圖召回,唔使 caption / 圖片 embedding。

▼
B

Inline 圖錨標記(W70–W71,ADR-0055)

合成答案時植入 [IMG#…] 標記,前端以 placeholder + 單一 ReactMarkdown 交織渲染 — 圖唔再全部堆落答案底部,而係跟住對應文字出現。copy / eval 路徑會自動 strip 標記。

▼
C

Section 級 → leaf 級錨定(W75 + W98–W99,ADR-0056 §Amendment)

無錨嘅輔助圖按 source_section 注入同章節錨點後(末尾堆 28–85% → 0%);W98 再加 section_anchor_nearest 以 doc_order 揀最近步驟做 leaf 級精準錨,W99 融入 P1_sop_imgdense preset 自動配對。全域 default 仍 OFF(B-03 等拍板)。

⑤企業級 RBAC 權限模型(W88–W95)

核心思想:權限喺入庫時收斂做 principal 清單蓋落 index,檢索時由 Azure AI Search 用字串比對 filter 直接 trim — 用戶睇唔到嘅文件根本唔會入候選池(唔係答案層遮掩)。群組展平到 group 級,成員變動零 re-stamp。

⑥統一整合層 — SharePoint 按需匯入(W100–W102,ADR-0070/0071/0072)

上半係用戶見到嘅 4 步 wizard(H7 重現 mockup);下半係 backend 資料流。兩個關鍵設計:① connector 只負責「攞文件 + 攞權限」,落地一律行既有 ingestion(圖 ③)— 實證 ingestion diff = 零;② SharePoint 權限經 get_principals 收斂到 group 級,直接復用 RBAC(圖 ⑤)嘅 allowed_principals 機制。

⑦三層可調配置 resolve(EKP 產品方向核心,ADR-0040~0063)

「唔同 KB / 文件用度身訂做配置」嘅落地機制:由粗到細逐層 override,越細層越贏;最終 effective config 決定圖 ② 每一階段嘅旋鈕。UI 三層對應:L1 自動(畫像配對)→ L2 一鍵 preset override → L3 逐旋鈕 + 試跑。

全域 Settings出廠值(ADR-0052 flip 後 baseline) ▶ 被覆蓋 per-KB KbConfigKB Settings tab · 一鍵套 preset ▶ 被覆蓋 per-doc DocConfig畫像自動配對 + 手動 override(手動贏) ▶ 被覆蓋 per-query override試跑 / A-B 草稿,唔 persist ＝ effective config驅動圖 ② 全部旋鈕

實務陷阱(已入 memory):做 A/B 實驗時,env 改全域會被目標 KB 嘅 per-KB config 鎖死 — 兩臂 effective config 一樣,差異全係 variance。正確做法係改 per-KB / per-doc / per-query 層。
